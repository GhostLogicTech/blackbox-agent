# blackbox-agent (rogue, system-monitor) Audit -- 2026-05-01

## Executive Summary

This is a standalone GhostLogic endpoint telemetry agent living outside the workspace at C:/Users/adam/system-monitor/blackbox-agent/, with its own git repo and its own C:/Program Files/GhostLogic/Agent/ install target. It collects 13 categories of host telemetry every 5s and POSTs to https://api.ghostlogic.tech with a tenant API key. It is the disabled v1.1.0-class endpoint monitor referenced in workspace memory feedback_ghostlogic_agent_disabled (2026-04-24). It is currently inert -- both Windows scheduled tasks (GhostLogicAgent, GhostLogicWatchdog) are absent from the schedule registry, the on-disk PID file is stale (Apr 18), no log activity since 2026-04-24 20:07.

The repo holds significant uncommitted post-disable hardening work (HMAC-SHA256 signing, drain rate limiting, --once smoke mode, --dry-run drain planner, --skip-backlog queue parking, abort-on-401/403/422 codes) that materially advances toward the disable-gate criteria. The 61,347-payload queue (~2.0 GiB) contains 56 hours of forensic telemetry from HP_Envy spanning 2026-04-22 to 2026-04-24 -- this is evidence-grade Blackbeard-class data, not waste, and must not be deleted.

**[F-002] HMAC secret reuse -- tenant_key serves as auth token AND signing key**
- Fingerprint: bba-hmac-secret-reuse-tenant-key-as-signing-key
- Category: security
- Evidence: agent/client.py:284-290 signs body with the same tenant_key it sends as Authorization Bearer. A server compromise, log leak, or MITM in demo_mode=true (cert verification disabled, client.py:34-39) recovers both halves of the trust model from one secret.
- Impact: HMAC-SHA256 signing exists but provides no integrity guarantee independent of tenant_key. A relay that captures the bearer token can forge signatures. The whole signing layer is theater unless secrets are split.
- Recommendation: Add a dedicated signing_secret field to config, separate from tenant_key. Either provisioned at registration time (return both from /api/v1/register) or rotated independently.
- evidence_note: Verified by direct code read; not exploited.

**[F-009] Queue directory growth is unbounded -- no cap, no rotation, no oldest-first eviction**
